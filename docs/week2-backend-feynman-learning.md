# 第 2 周：AI 应用后端能力强化（费曼学习法）

目标：把 AgentFlow AI 从“能演示的 demo”升级成“更像真实服务的后端工程”。

本周关键词：

- 分层：路由只负责 HTTP，Service 负责业务，Repository 负责数据。
- 契约：请求、响应、错误、SSE 事件都要有稳定格式。
- 韧性：超时、重试、降级、错误边界要可控。
- 可观测：请求 ID、日志、模型调用耗时、token 与成本要能追踪。
- 会话：聊天不只是一次请求，而是一串有上下文的消息。

## 费曼学习法怎么用

每天按 4 步走：

1. 先学一个概念。
2. 用最简单的话讲给“完全不懂后端的人”听。
3. 找出自己讲不清楚的地方，回代码和文档补洞。
4. 用项目里的真实接口改一小块，再写一段 30 秒复述。

每天输出 3 个东西：

- 一段“小白解释”。
- 一个项目改造点。
- 一个面试口述答案。

判断是否学会：

- 你能不用术语解释它。
- 你能指出它在当前项目里的文件位置。
- 你能说清楚“没有它会出什么问题”。

## Day 1：后端分层与项目结构

学习目标：

- 理解 FastAPI 项目为什么要分层。
- 明确 `api`、`schemas`、`services`、`core` 各自职责。
- 为后续 repository、middleware、errors 等模块预留位置。

当前项目落点：

- `backend/app/api/chat.py`
- `backend/app/api/requirements.py`
- `backend/app/services/llm_service.py`
- `backend/app/schemas/`
- `backend/app/core/config.py`

费曼解释：

> 后端分层就像餐厅分工。服务员负责接单，对应 API 路由；厨师负责做菜，对应 Service；仓库管理员负责取食材，对应 Repository；菜单和价格表对应 Schema 和 Config。一个人什么都干，餐厅小的时候没问题，人一多就会乱。

开发任务：

- 梳理当前后端目录职责，写进 README 或本文件备注。
- 不急着大重构，先确认每个函数是否放在正确层。
- 如果新增会话存储，优先放到 `backend/app/services/` 或新增 `backend/app/repositories/`。

面试口述：

> FastAPI 路由层应该尽量薄，只处理请求参数、状态码和响应格式；业务逻辑放在 service；数据读写放在 repository；Pydantic schema 负责输入输出契约。这样做的好处是接口变了不影响模型调用，存储方式变了也不影响路由，单测和 mock 都更容易。

自测问题：

- `LLMService` 现在承担了哪些职责？
- 如果以后把会话存在数据库，应该改路由还是新增 repository？
- 路由函数里出现大量模型调用细节，说明什么问题？

## Day 2：统一错误响应

学习目标：

- 区分业务错误、参数错误、外部服务错误、系统错误。
- 理解为什么不能把所有异常都直接丢给前端。
- 设计统一错误格式。

建议错误格式：

```json
{
  "error": {
    "code": "LLM_TIMEOUT",
    "message": "模型服务响应超时，请稍后重试",
    "request_id": "req_20260514_xxx"
  }
}
```

费曼解释：

> 错误响应就像快递异常通知。不能只写“失败了”，用户和开发都不知道是地址错、快递员没来、仓库没货，还是系统坏了。统一错误格式就是给所有失败情况贴上固定标签。

开发任务：

- 先列出当前项目可能出现的错误：
  - 没有配置 API Key。
  - 模型供应商报错。
  - 模型输出不是合法 JSON。
  - SSE 流中断。
  - 用户请求参数为空。
- 后续可新增 `backend/app/core/errors.py` 管理错误码和异常转换。

面试口述：

> 我会把错误分成客户端错误、业务错误、外部依赖错误和系统错误。对前端返回稳定的 `code/message/request_id`，对日志记录更完整的堆栈和供应商错误。这样用户看到的是可理解的信息，开发能用 request_id 追到具体日志。

自测问题：

- 为什么不能把 OpenAI/DeepSeek 原始错误完整返回给前端？
- SSE 已经开始返回 token 后，中途报错怎么通知前端？
- `422`、`502`、`503` 分别适合什么场景？

## Day 3：请求 ID 与日志

学习目标：

- 理解请求 ID 是排查问题的主线。
- 掌握请求日志、业务日志、模型调用日志的区别。
- 让日志能回答：谁在什么时候请求了什么，耗时多久，失败在哪里。

费曼解释：

> 请求 ID 就像一次外卖订单号。用户说“我的订单有问题”，客服不能靠猜，要用订单号查下单、出餐、配送、签收每一步。后端请求也是一样，一个 request_id 串起路由、模型调用、工具调用和错误。

建议日志字段：

- `request_id`
- `method`
- `path`
- `status_code`
- `duration_ms`
- `model`
- `mocked`
- `tool_name`
- `error_code`

开发任务：

- 后续可新增 middleware，在每个请求进入时生成 `request_id`。
- 响应 header 可带 `X-Request-ID`。
- 模型调用日志不要记录完整敏感输入，最多记录 preview、长度、耗时和状态。

面试口述：

> 我会用 middleware 给每个请求生成 request_id，并把它放进日志上下文和响应 header。普通请求记录方法、路径、状态码、耗时；模型调用额外记录模型名、是否 mock、token 用量、错误类型。敏感内容不直接打全量日志，只做截断或脱敏。

自测问题：

- 为什么日志里不应该完整记录用户 prompt？
- request_id 应该在前端可见吗？
- 模型调用失败时，日志至少要保留哪些信息？

## Day 4：模型调用超时、重试与降级

学习目标：

- 理解 AI 接口比普通接口更需要超时控制。
- 区分“可以重试”和“不应该重试”的错误。
- 设计 mock、降级、失败提示策略。

费曼解释：

> 调模型像打电话给外包专家。你不能一直等他接电话，要设定等待时间；如果是占线，可以过几秒再打一次；如果号码错了，重试没有意义；如果专家今天不可用，就告诉用户当前只能看缓存或稍后再试。

适合重试的情况：

- 网络临时失败。
- `429` 限流，但需要退避等待。
- `5xx` 供应商服务异常。
- 连接中断。

不适合盲目重试的情况：

- API Key 错误。
- 请求参数非法。
- 余额不足。
- 内容安全策略拒绝。

开发任务：

- 给模型调用加超时配置，例如 `LLM_TIMEOUT_SECONDS`。
- 重试要限制次数，例如最多 2 次。
- SSE 场景下要谨慎：已经输出了一半内容时，不要简单自动重试并拼接到同一条消息。

面试口述：

> 模型接口必须设置超时，否则一个慢请求会占住连接和资源。重试只针对临时性错误，比如网络抖动、限流、供应商 5xx，并且要有最大次数和退避策略。对于鉴权失败、参数错误、余额不足这类确定性错误，不重试，直接返回明确错误。流式输出如果已经开始吐 token，中途失败通常交给前端提示重试，而不是后端自动拼接第二次结果。

自测问题：

- 普通 HTTP 补全和 SSE 流式补全的重试策略有什么不同？
- 供应商 429 应该立刻重试吗？
- 如果模型超时，前端应该显示什么？

## Day 5：Token 与成本统计

学习目标：

- 理解 token 是模型计费和上下文长度的基本单位。
- 设计每次模型调用的 usage 记录。
- 能说清楚成本统计对企业 AI 应用的意义。

费曼解释：

> Token 成本就像水电表。每次模型调用都在用“水电”，不装表的时候感觉只是问答，等账单来了才发现哪个功能最贵。统计 token 是为了知道钱花在哪里，以及哪里需要优化。

建议记录字段：

- `provider`
- `model`
- `prompt_tokens`
- `completion_tokens`
- `total_tokens`
- `estimated_cost`
- `request_id`
- `conversation_id`
- `created_at`

开发任务：

- 如果供应商响应有 `usage`，优先读取真实 usage。
- 如果流式接口最后没有 usage，可先记录 `null`，不要假装有精确数据。
- 成本单价先用配置或常量，后续再做模型价格表。

面试口述：

> 我会在模型服务层统一采集 usage，因为所有模型调用都会经过这一层。非流式接口通常能直接拿到 prompt/completion tokens；流式接口要看供应商是否在最终 chunk 返回 usage。成本估算不要散落在业务代码里，而是按 provider/model 做配置化价格表，便于后续统计每个用户、会话、功能的成本。

自测问题：

- token 统计为什么不应该放在前端？
- 流式接口没有返回 usage 怎么办？
- 企业为什么关心单用户、单功能、单会话成本？

## Day 6：简单会话存储

学习目标：

- 理解一次聊天请求和一个会话的区别。
- 设计最小会话模型。
- 明确内存存储、文件存储、数据库存储的边界。

费曼解释：

> 一次聊天请求像一句话，会话像一整段对话。没有会话存储，刷新页面就像失忆；有了会话 ID，后端知道这次回答应该接着哪段上下文说。

最小数据结构：

```json
{
  "conversation_id": "conv_xxx",
  "messages": [
    { "role": "user", "content": "..." },
    { "role": "assistant", "content": "..." }
  ],
  "created_at": "2026-05-14T10:00:00+08:00",
  "updated_at": "2026-05-14T10:03:00+08:00"
}
```

开发任务：

- 先做内存版会话存储即可，明确它重启会丢。
- API 可逐步支持 `conversation_id`。
- 存历史时要限制最大消息数或最大 token，避免上下文无限增长。

面试口述：

> 聊天系统需要会话 ID 来管理上下文。最小实现可以用内存字典保存 conversation_id 到 messages 的映射，适合 demo；生产环境要换成数据库或 Redis，并加用户隔离、过期时间和上下文裁剪。发送给模型的历史不能无限增长，需要按轮数、token 或摘要策略压缩。

自测问题：

- 为什么不能把全部历史无限传给模型？
- 内存会话存储有什么风险？
- conversation_id 应该由前端生成还是后端生成？

## Day 7：整合、复盘与 2 分钟讲解

学习目标：

- 把一周内容串成“工程化后端”的故事。
- 能解释 AI Chat v2 比 v1 强在哪里。
- 为第 3 周 RAG 做准备。

本周复盘模板：

```text
第 2 周我把 AgentFlow AI 的后端从 demo 往工程化服务推进。

第一，做分层，让路由、业务和数据职责更清晰。
第二，做统一错误，让前端和开发都能稳定理解失败原因。
第三，做请求 ID 和日志，让一次请求从入口到模型调用都能追踪。
第四，做超时、重试和降级，让模型供应商不稳定时系统不至于卡死。
第五，设计 token 和成本统计，让 AI 功能具备企业级成本意识。
第六，加入会话存储，让聊天从单次请求变成可持续对话。

这些能力不直接改变页面外观，但决定了 AI 应用能不能被真实团队长期使用。
```

2 分钟面试讲法：

> 第 1 周我做的是 AI Chat 最小闭环：FastAPI、Next.js、SSE、结构化输出和 Tool Calling。第 2 周我重点补后端工程能力，因为真实 AI 应用最大的问题不是“能不能调通模型”，而是能不能稳定、可追踪、可控成本地运行。
>
> 我会把 FastAPI 路由保持轻量，模型调用集中在 service 层；用 Pydantic 固定请求和响应契约；用统一错误格式让前端能稳定处理失败；用 request_id 把 HTTP 请求、模型调用、工具调用和日志串起来；对模型接口设置超时和有限重试；再记录 usage，用于 token 和成本统计。最后加一个简单会话存储，让聊天能带上下文。
>
> 这样 AgentFlow AI 就不只是一个演示页面，而是具备工程化后端雏形的 AI 应用。
